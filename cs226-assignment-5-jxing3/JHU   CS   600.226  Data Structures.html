Assignment 5: Linked Lists

Out on: October 4, 2013
Due by: October 11, 2013 before 10:00 pm
Collaboration: None
Grading: Packaging 10%, Style 10% (where applicable), Testing 20% (where applicable), Performance 10% (where applicable), Functionality 50% (where applicable)

Overview

The fifth assignment is mostly about the list implementation using sentinels that we discussed in lecture. I was going to include an application as well, but seeing as you have the midterm to deal with, I wanted to make sure that this is a shorter assignment instead. You'll need to grab this archive for interfaces and exception classes. Maybe you want to take a quick tour of the files before you read on?

Problem 1: List Implementation (90%)

Your first task for this assignment is to write a generic linked list class SentinelList<T> that implements all of the interfaces we designed in lecture: List<T>, Position<T>, and Iterator<T>. Unlike the NodeList<T> implementation that we hacked in lecture, your SentinelList<T> will use the sentinel nodes technique to further simplify the code for most list operations. You will still be able to reuse a large part of the code from lecture, and you're allowed to do so. Please remember that your implementations of the Position<T> and Iterator<T> interfaces should both be nested inside your SentinelList<T> class.

You should provide a toString() method in addition to the methods required by the List<T> interface. A new list into which 1, 2, and 3 were inserted using insertBack() should print as [1, 2, 3] while an empty list should print as [].

Your SentinelList<T> class should come with a full suite of JUnit 4 test cases using theories in a file TestList.java. Be sure to test all methods and be sure to test all exceptions for error situations as well. You really need to write rock-solid test cases for this one. I held back in lecture and provided just a few examples, there is much more that needs to be tested! Note that I even changed the grading criteria to reflect that your test cases are very important for this assignment!

Please pay extra attention to avoiding duplicated code! For example, you should have a private method that validates incoming positions, you should not repeat the validation code eight separate times in each method that needs to perform the validation. Also many of the insertX and removeX methods will rely on code that is incredibly similar, try to factor that code into separate private methods as well!

Problem 2: List Reflection (10%)

Your second task for this assignment is to write up a reflection on the design we've come up with for our list abstractions. Look at the interfaces, their purpose and interrelation, look at our implementation techniques, for example the sentinel idea, etc. Feel free to use other designs you know about (for example from the Java class libraries, from a text book, from libraries of other languages you know, etc.) to inform your critique. Your reflection should go into your README file. We won't be hurt if you "diss" our design, just be sure that you're making a good case that you can back up with respectable sources!

Deliverables

Please turn in a gzip compressed tarball of your assignment; the filename should be cs226-assignment-5-name.tar.gz with name replaced by the first part of the email address you used to register on Piazza (so I would use cs226-assignment-5-phf.tar.gz). The tarball should contain no derived files whatsoever (i.e. no .class files, no .html files, etc.), but allow building all derived files. Include a README file that briefly explains what your programs do and contains any other notes you want us to check out before grading; your answers to written problems should be in this file as well.

Grading

For reference, here is a short explanation of the grading criteria; some of the criteria don't apply to all problems, and not all of the criteria are used on all assignments. Packaging refers to the proper organization of the stuff you hand in, following both the guidelines for Deliverables above as well as the general submission instructions for assignments. Style refers to Java programming style, including things like consistent indentation, appropriate identifier names, useful comments, suitable javadoc documentation, etc. as enforced by checkstyle with our JHU configuration file. Style also includes proper modularization of your code (into interfaces, classes, methods, and using public, protected, and private appropriately). Simple, clean, readable code is what you should be aiming for. Testing refers to proper unit tests for all of the data structure classes you developed for this assignment, using the JUnit 4 framework. Make sure you test all (implied) axioms that you can think of and all exception conditions that are relevant. Performance refers to how fast/with how little memory your program can produce the required results compared to other submissions. Design refers to proper modularization (classes, public, private, etc.) and the proper choice of algorithms and data structures. Functionality refers to your programs being able to do what they should according to the specification given above; if the specification is ambiguous and you had to make a certain choice, defend that choice in your README file.

If your programs cannot be built you will get no points whatsoever. If your programs cannot be built without warnings using javac -Xlint:all we will take off 10% (except if you document a very good reason). If your programs fail miserably even once, i.e. terminate with an exception of any kind, we will take off 10% (however we'll also take those 10% off if you're trying to be smart by wrapping your whole program into a universal try-catch). Finally, make sure to include your name and email address in every file you turn in (well, in every file for which it makes sense anyway)!
